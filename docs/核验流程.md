# 本地核验流程

追加功能、修 BUG、改启动流程或准备发布前，必须先在本地跑通核验。

## 必跑命令

生成发布包并核验：

```powershell
powershell -ExecutionPolicy Bypass -File scripts\build_release.ps1
```

只核验现有发布包：

```powershell
powershell -ExecutionPolicy Bypass -File scripts\verify_release.ps1
```

## 核验内容

`verify_release.ps1` 会检查：

- Python 语法是否通过。
- 核心模块是否可导入。
- README、AGENTS、CHANGELOG、VERSION、启动脚本、Release 工作流是否存在。
- 发布包 `releases/codex-bubble-vX.Y.Z.zip` 是否存在。
- 发布包是否包含必要源码、文档和预览图。
- 发布包是否误包含 `.git/`、`data/`、`logs/`、`__pycache__/`、`*.log` 或 `codex_usage_data.json`。
- 开发缓存和临时文件是否被 `.gitignore` 忽略，并从发布包中排除。

## 发布纪律

- 没有跑通 `scripts\verify_release.ps1`，不得打版本 tag。
- 没有跑通 `scripts\verify_release.ps1`，不得推送 GitHub Release。
- 发现问题先修复，再重新构建和核验。
- 核验通过后，再提交、打 tag、推送到 GitHub。
- 如果临时文件、缓存、日志或运行时数据进入 Git 状态或发布包，必须先修复 `.gitignore` 和构建脚本。

## 自动化验证不了时

有些内容无法只靠脚本验证，例如：

- 悬浮球是否真的显示在桌面上。
- 拖动、展开、右键菜单等交互是否符合预期。
- 多屏环境下是否出现在正确屏幕。
- 视觉样式是否符合预期。

遇到这些情况时，必须启动本地软件，请用户现场确认。用户确认前，不得正式打包上传 GitHub Release。

## 界面变化要求

只要界面、交互或外观发生变化，必须同步做三件事：

1. 本地运行软件验证。
2. 截取新的软件预览图，保存到 `docs/assets/`。
3. 更新 `README.md` 中的预览图或说明。
