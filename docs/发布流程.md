# 发布流程

本项目用 Git tag + GitHub Release 做版本发布。

## 版本号规则

使用语义化版本号：

- `0.1.0`：第一个公开预览版。
- `0.x.0`：新增功能或明显体验改进。
- `0.x.y`：问题修复、文档修正或小范围兼容改动。
- `1.0.0`：功能和安装体验稳定后发布正式版。

## 发布前检查

1. 更新 `VERSION`。
2. 更新 `CHANGELOG.md`。
3. 本地构建并核验：

```powershell
powershell -ExecutionPolicy Bypass -File scripts\build_release.ps1
```

4. 确认 `scripts\verify_release.ps1` 输出核验通过。
5. 如果改动来自 PR，确认 `PR Review Gate` 已通过。
6. 提交代码，不提交本地生成的安装器或 zip。
7. 创建并推送版本 tag。

发布前必须确认安装器内部 payload 不包含：

- `.git/`
- `data/`
- `logs/`
- `__pycache__/`
- `*.log`
- 本机生成的 `codex_usage_data.json`

## GitHub Release 内容

Release 标题格式：

```text
Codex 额度悬浮球 vX.Y.Z
```

Release 附件：

```text
codex-bubble-setup-vX.Y.Z.exe
```

Release 说明从 `CHANGELOG.md` 对应版本复制。

## 自动发布

仓库包含 `.github/workflows/release.yml`。推送 `v*` 标签后，GitHub Actions 会先运行 `scripts\build_release.ps1` 构建安装器，再自动创建 Release，并上传对应的安装器：

```text
releases/codex-bubble-setup-vX.Y.Z.exe
```

注意：没有本地核验通过，不得推送 tag 触发自动发布。

## PR 自动审核

仓库包含 `.github/workflows/pr-review.yml`。每个指向 `main` 的 PR 会自动执行：

- `git diff --check`：检查 PR diff 中的空白错误。
- `scripts\build_release.ps1`：构建安装器并复用完整发布验证。
- 安装器安全检查：确认 payload 不包含 `.git/`、`data/`、`logs/`、`__pycache__/`、日志或本机额度数据。
- 安装/卸载烟测：安装到临时目录，再调用卸载脚本确认目录会被移除。
- 上传 7 天有效的安装器 artifact，方便人工下载做界面或交互检查。

这个自动审核只证明工程门禁通过；如果改动影响界面、交互或用户感知，仍然需要人工运行确认并更新 README 截图。
